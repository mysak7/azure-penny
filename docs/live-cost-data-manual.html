Live Cost Data Manual

How azure-penny collects, merges, and displays live resource costs

Introduction

The Live Resources page (/live) answers one question: what am I spending right now, and on what? It combines two independent data sources into a single table — the historical billing record from Azure Cost Management exports, and the real-time resource inventory from Azure Resource Manager.

Key insight: Cost Management exports are always 1–2 days behind. The live ARM inventory is current to the minute. azure-penny correlates them by Resource ID so every live resource shows its actual billed cost where available, and falls back to price-list estimates for resources too new to appear in exports yet.

Two Data Pipelines

Every number you see on the Live Resources page comes from one of two independent pipelines that run in parallel and are then merged.

Pipeline 1 — Historical billing (cost exports)
Azure Cost Management → daily CSV → Blob Storage
cost-exports/ → download → pandas DataFrame → 1h cache → C_COST per resource
Pipeline 2 — Live inventory (ARM)
Azure Resource Manager → REST API → list_resources() → VM states, SKUs, IPs → resource list → 15min cache → live inventory
Merge
C_COST per resource + live inventory → join on Resource ID → enriched list → /api/live-resources

Export Schedule

Terraform provisions a daily Cost Management export at billing profile scope using the azapi_resource provider (the standard azurerm provider does not support MCA/billing profile scope).

SettingValueReason
ScheduleDailyFresh data every morning

TypeActualCostReal charges, not amortised

Destination pathpenny/{date-range}/{guid}/filename.csvAzure's default export folder structure

Lag warning: Today's charges never appear in today's export. Cost Management closes and exports the previous day's data roughly 1–2 hours after midnight UTC. The app always displays the most recently available export, which will be yesterday's or earlier.

Blob Discovery

On every cache miss, storage.py:_discover_latest_blobs() scans the cost-exports container and selects exactly one file to load — the most recently modified file in the most recent date-range folder.

Discovery algorithm

List all blobs → Extract YYYYMMDD-YYYYMMDD segments → Sort descending (newest first) → Pick latest folder with .csv / .parquet → If multiple files in folder → pick newest by last_modified → Single blob path returned

Why only one file?

Azure Cost Management exports are cumulative: each daily export contains all charges from the start of the billing month to that day. Loading multiple files would double-count costs. The app intentionally loads only the newest file — which contains the most complete data.

Fallback behaviour

If no YYYYMMDD-YYYYMMDD folder pattern is found (e.g. a manually uploaded file), the discovery falls back to picking the single newest .parquet or .csv/.csv.gz file in the entire container by last_modified timestamp.

Supported file formats

  • .parquet — read via pd.read_parquet(engine="pyarrow")
  • .csv — read via pd.read_csv()
  • .csv.gz — read via pd.read_csv(compression="gzip")

The raw bytes are streamed into memory via io.BytesIO — no temporary disk files are used.

Post-mapping transformations

  • C_COST is coerced to float64; unparseable values become 0.0.
  • C_DATE is parsed as datetime and reformatted to YYYY-MM-DD string.
  • C_NAME (resource group) is lowercased for case-insensitive matching.

In-Memory Cache (Cost Exports)

Loading a cost export CSV from Blob Storage takes several seconds. To avoid re-downloading on every API request, the DataFrame is held in a process-level in-memory cache protected by an asyncio.Lock.

PropertyValue
TTL (time-to-live)CACHE_TTL_SECONDS env var, default 3600 s (1 hour)
Concurrency guardasyncio.Lock — only one reload runs at a time; other requests wait
Cache miss behaviourDownload blob → parse → apply column map → store; returns empty DataFrame if no blobs found
Manual invalidationPOST /api/reload or GET /costs/refresh — clears _cache dict
Storage locationModule-level _cache dict in storage.py — lives for the lifetime of the uvicorn process
Scale-to-zero note: When the Container App cold-starts (first request after being idle), the cache is empty. The first API call will trigger a fresh download from Blob Storage before returning. Subsequent calls within the TTL window are fast.

ARM Inventory

live_resources.py:_fetch_resource_inventory() calls the Azure Resource Manager REST API to list every resource in the subscription. This uses the azure-mgmt-resource SDK, authenticated via DefaultAzureCredential with the Container App's user-assigned managed identity.

What gets listed

ResourceManagementClient.resources.list() returns all resources across all resource groups and all regions in the configured subscription. Deallocated and stopped VMs are excluded from the returned list (they contribute zero ongoing cost and clutter the view).

Required RBAC

The managed identity needs at minimum Reader on the subscription to list resources. It is currently assigned Contributor (which also enables the resource deletion feature). Without this role, the live inventory returns an empty list.

VM Power States

For every Microsoft.Compute/virtualMachines resource found, a second API call is made to ComputeManagementClient.virtual_machines.get(rg, name, expand="instanceView") to fetch the power state. This is the only way to distinguish a running VM from a deallocated one.

Power stateDisplayed asIncluded in list?
VM running● RunningYes
VM stopped● StoppedNo — excluded
VM deallocated● DeallocatedNo — excluded
Unknown (API error)● UnknownYes (kept to be safe)

For VMs, the inventory also captures:

  • VM size (e.g. Standard_D2s_v3) — used for price lookups
  • Is spot (boolean) — triggers spot pricing instead of on-demand
  • Private IP — via the first NIC's first IP configuration
  • Instance count — for VM Scale Sets, the SKU capacity field

Azure Files Premium Shares

For every Microsoft.Storage/storageAccounts resource found, an additional call is made to StorageManagementClient.file_shares.list(rg, account_name) to enumerate premium file shares. Each share is added as a synthetic resource to the inventory with a pre-computed monthly estimate.

The cost estimate is: provisioned_quota_GiB × price_per_GiB_per_month. The per-GiB price is looked up from the Azure Retail Prices API (with a regional fallback table). This is a provisioned capacity estimate — the share is charged for its quota regardless of actual usage.

Storage account suppression: When a storage account has live file shares in the inventory, the storage account's own cost row is suppressed (cost_source = "in_shares") to avoid double-counting. The total cost for the account is shown through the individual share rows instead.

ARM Cache (Live Inventory)

ARM API calls are expensive — listing all resources in a large subscription can take 5–30 seconds. The live inventory is cached separately from the cost export cache.

PropertyValue
TTLLIVE_CACHE_TTL_SECONDS env var, default 900 s (15 minutes)
Concurrency guardSeparate asyncio.Lock (_live_lock) from the export cache lock
Cache miss behaviourCall ARM → fetch VM power states → enumerate file shares → store
Manual invalidationPOST /api/live-reload — clears _live_cache dict, then re-fetches
ExecutorAll blocking SDK calls run in asyncio.run_in_executor(None, ...) (thread pool) to avoid blocking the event loop

The two caches (_cache in storage.py and _live_cache in live_resources.py) are independent. A cost export reload does not flush the ARM inventory, and vice versa.

Merging the Two Pipelines

live_resources.py:_get_live_data() runs after both caches are populated. It joins the ARM inventory with the cost DataFrame on Resource ID.

Resource ID normalisation

ARM resource IDs and Cost Management resource IDs are not always identical — Cost Management sometimes includes sub-resource paths (e.g. .../fileServices/default). Before joining, both sides are:

  1. Lowercased (ARM IDs are case-insensitive)
  2. Stripped of sub-resource suffixes via _normalize_rid() regex

30-day window

Only the last 30 days of cost data are used for the live view. The cost export file contains the full billing month, but the live page is about current spend, not historical totals. 

cutoff = (date.today() - timedelta(days=30)).strftime("%Y-%m-%d")
mdf = df[df["C_DATE"] >= cutoff]

Per-resource aggregation

After filtering to 30 days, the DataFrame is grouped by (normalised) Resource ID:

  • cost_by_id — total cost over the 30-day window, per resource
  • days_by_id — number of distinct billing days the resource appeared
  • hours_by_id — total usage quantity (hours) for VMs
  • last_day_cost_by_id — cost on the most recent export day (for new-resource detection)

Projection logic

The monthly_cost field on each enriched resource is the estimated monthly spend. The projection method used depends on how many days of data are available for that resource:

Days of dataMethodSource tag
≥ 28 days Use the 30-day billed total directly — no projection needed export
< 28 days, storage account with no live shares Show billed total as-is; projecting storage is misleading (variable usage) export_period
< 28 days, VM with usage hours (total_cost / total_hours) × 24 × 30 export_hours
< 28 days, general resource (total_cost / export_days) × 30 export_projected
< 28 days, storage account with shares, last-day spike last_day_cost × 30 (when larger than window projection) export_last_day

VM price override

For active VMs (not deallocated), the projection from export data is overridden by a live price-API lookup. This gives a more accurate current estimate even when the resource is newer than the export window:

  • Spot VMs: spot_price_per_hour × 24 × 30 × instance_count
  • On-demand VMs: on_demand_price_per_hour × 24 × 30 × instance_count

If no export data exists for the VM at all, this price-API estimate is the only cost shown.

Price Calculation: Full Decision Tree

Every resource goes through a two-phase calculation to produce its monthly_cost. Phase 1 (VM override) always wins for active VMs; Phase 2 (export) handles everything else.

For each live ARM resource…
Has vm_size + location and is active? → yes → Phase 1: VM Live Price API → price_table / spot_rate
no → Phase 2: Cost Export → export has data? yes → project to monthly
no → is a file share? → yes → files_rate (quota × $/GiB)
no → null — shown as —

Why VMs Always Have a Price

For any resource with a non-empty vm_size field (populated during ARM inventory for Microsoft.Compute/virtualMachines and Microsoft.Compute/virtualMachineScaleSets), the code calls the Azure Retail Prices API unconditionally — even if the cost export already has data for that VM. This override exists because:

  • Export data is 1–2 days stale; a VM started today has no export entry yet.
  • The Retail API returns the current on-demand or spot rate — a better estimate of ongoing spend than a partial-period average.
  • Spot prices fluctuate; the API always returns today's market price.

VM price lookup — spot vs on-demand

VM priority = Spot? → yes → _fetch_spot_price(vm_size, region) → spot_rate
VM priority = Regular? → yes → _fetch_ondemand_price(vm_size, region) → price_table

Three-tier price fallback (applies to both functions)

  1. Azure Retail Prices API — prices.azure.com/api/retail/prices, filtered by armRegionName, armSkuName, and priceType eq 'Consumption'. Spot lookups add a secondary filter matching "spot" in the returned SKU name. Results are cached in-process for the app lifetime.
  2. Hardcoded table — _SPOT_PRICES / _ONDEMAND_PRICES in live_resources.py, covering common Dsv3, Esv3, and Fsv7 sizes across major regions. Used only when the API call fails or times out.
  3. 40% estimate (spot only) — if neither the API nor the table has a spot price, the on-demand price × 0.40 is used (a rough industry average spot discount).

The final monthly estimate formula:

# on-demand VM (or VMSS)
monthly_cost = on_demand_$/hr × 24 × 30 × instance_count

# spot VM (or VMSS)
monthly_cost = spot_$/hr × 24 × 30 × instance_count
Price is not actual cost: The Retail Prices API returns list prices, not your negotiated rate. Reserved Instances, Azure Hybrid Benefit (AHUB), and Dev/Test subscriptions all result in lower actual charges. Once the export contains data for the VM, the export / export_hours sources reflect real billed amounts. The live API price is an upper-bound estimate.

Why Other Resources Show —

Non-VM resources (storage accounts, VNets, load balancers, databases, Container Apps, etc.) have no vm_size to query the price API with. Their cost can only come from the cost export. A resource displays — (null cost) when:

  • It was created after the most recent export was generated (new resource, not yet billed).
  • It exists in ARM but generates no direct Azure charges — e.g. a VNet or a DNS zone.
  • It costs money but under a parent resource's billing line, without a separate resource ID row in the export.
  • The export has no data for the subscription or scope the managed identity can read.

The known_has_cost flag and Hide no-cost behaviour

Each resource carries a boolean known_has_cost set to true if its ARM resource type appears in _RTYPE_CATEGORY (classified as vm, storage, network, database, or container). Types not in that dictionary fall into the other category and get known_has_cost = false.

The Hide no-cost checkbox (enabled by default) uses this flag to decide what to hide: 

// hides a resource only when ALL THREE conditions are true
if (hideFree && cost === null && !r.known_has_cost) return false;

Live Resources Page

The /live page renders a sortable, filterable table of all active ARM resources enriched with cost data. The page makes two API calls in parallel on load:

  • GET /api/live-resources — full enriched resource list
  • GET /api/live-services — 30-day cost totals aggregated by Azure service (for the Billed 30d stat)

Refreshing Data

Automatic refresh

The page does not auto-refresh. Data shown is whatever was cached when you loaded the page.

Manual refresh options

ActionWhat it doesHow to trigger
Refresh inventory Clears the ARM cache (_live_cache) and re-fetches all resources from ARM. Cost export cache is untouched. Click ↻ Refresh inventory button in the status bar
Reload cost export Clears the export DataFrame cache and re-downloads the latest blob from Blob Storage. POST /api/reload or GET /costs/refresh (no UI button on the live page)
Full reload Reload cost export first, then refresh inventory. Call both endpoints manually
Cache TTL reminder: The cost export cache (1h) and ARM cache (15min) expire independently. You may see fresh ARM data with stale cost numbers, or vice versa, if only one cache was refreshed.

Pricing Fallbacks

Price lookups for VMs and Azure Files shares follow a three-tier fallback chain to handle API unavailability.

VM prices (spot and on-demand)

Azure Retail Prices API
prices.azure.com → if fails → Hardcoded _SPOT_PRICES / _ONDEMAND_PRICES tables
in live_resources.py → if VM size not in table → 40% of on-demand price
(industry average for spot) → if still no price → null — show —

Azure Files Premium prices

Azure Retail Prices API
Storage filter → if fails → Hardcoded _FILES_PREMIUM_PRICES table
default: $0.21/GiB/mo

Price caching

All price lookups are cached in-process for the lifetime of the Container App instance (module-level _SPOT_PRICE_CACHE and _FILES_PRICE_CACHE dicts with no TTL). Prices are not refreshed until the app restarts. Given the Container App scales to zero daily, prices are effectively re-fetched on every cold start.

Staleness risk: The hardcoded fallback tables in live_resources.py contain spot and on-demand prices that were accurate at the time the code was written. Azure spot prices fluctuate. If the Retail Prices API is consistently reachable, this is not a problem — the API result always takes precedence.

Known Limitations

LimitationImpactWorkaround
Cost data is 1–2 days old Today's charges are never shown; yesterday's appear only after the export runs (~midnight UTC + 2h) Check Azure Cost Management portal for same-day estimates
Single subscription only Resources and costs from other subscriptions are invisible Deploy a separate azure-penny instance per subscription
Deleted resources show up in "Deleted/hidden" cost You cannot see which specific deleted resources generated the orphaned cost Query Cost Management directly with ResourceId filter for deleted IDs
VM prices are retail, not negotiated If you have Azure Reserved Instances or AHUB (Azure Hybrid Benefit), actual cost will be lower Use export-based cost sources (export, export_projected) which reflect actual charges
File share estimate uses provisioned quota, not actual usage Estimate may be higher than billed if the share is mostly empty Premium file shares bill for provisioned capacity; this is correct for Premium tier
AKS node pools are VMSS resources, not individual VMs Node-level power states and IPs are not fetched The VMSS row shows aggregate cost via instance count × price
Hardcoded price tables cover only common VM sizes Uncommon VM sizes (e.g. Standard_ND GPU series) may show — if the Retail API is down The live Retail Prices API covers all sizes; only the fallback tables are limited
Cold start latency First request after scale-to-zero idle period triggers both cache loads simultaneously — response may take 15–30s Open /health to warm the app; costs load lazily on first API request

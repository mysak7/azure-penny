Three Identity Layers
Each solves a different problem · none require stored passwords
🐙
1 — GitHub Actions SP
CI/CD identity
OIDC federated service principal used by GitHub Actions to run terraform apply and push Docker images to ACR.

Requires Azure RBAC roles on the subscription and a Microsoft Graph application permission.
Managed by: manual bootstrap (once)
🔑
2 — User-Assigned Managed Identity
Runtime identity
Attached to the Container App. Used by the Python app at runtime to read Blob Storage, pull from ACR, and call Azure Resource Manager.

DefaultAzureCredential picks it up automatically via the IMDS endpoint — no code changes needed between local dev and production.
Managed by: Terraform (identity.tf, acr.tf)
🛡️
3 — Easy Auth App Registration
UI access guard
An Entra ID app registration wired into Container Apps Easy Auth. Any browser request to the dashboard is redirected to a Microsoft login page first.

app_role_assignment_required = true means only explicitly whitelisted users can sign in.
Managed by: Terraform (auth.tf)

⚠️ Bootstrap: Graph Permission
The DelegatedPermissionGrant.ReadWrite.All permission cannot be granted by Terraform — it's the same principal Terraform runs as. A Global Admin or Privileged Role Admin must run this once.
# Get CICD SP object ID by display name SP_OID=$(az ad sp show \ --display-name "azure-penny-github-actions" \ --query id -o tsv) # Get Microsoft Graph SP object ID in this tenant MSGRAPH_OID=$(az ad sp show \ --id 00000003-0000-0000-c000-000000000000 \ --query id -o tsv) # Assign the app role (= grant admin consent) az rest --method POST \ --uri "https://graph.microsoft.com/v1.0/\ servicePrincipals/$SP_OID/appRoleAssignments" \ --headers "Content-Type=application/json" \ --body "{ \"principalId\": \"$SP_OID\", \"resourceId\": \"$MSGRAPH_OID\", \"appRoleId\": \"8e8e4742-1d95-4f68-9d56-6ee75648c72a\" }"
App role ID 8e8e4742-…c72a is stable and well-known for this permission across all tenants.

Why User-Assigned, not System-Assigned?
  • 1Survives Container App replacement — the identity is independent of the app resource lifecycle.
  • 2Role assignments can reference the MI before the app exists — no ordering dependency in Terraform.
  • 3The same identity can be re-used or referenced in future resources (e.g. AKS, Functions).

The Admin Consent Problem
Why azuread_service_principal_delegated_permission_grant fails in CI and how to fix it
What goes wrong
  • 1Terraform runs as the GitHub Actions service principal (azure-penny-github-actions).
  • 2azuread_service_principal_delegated_permission_grant calls the Graph API to create a tenant-wide admin consent record.
  • 3Graph requires the caller to hold DelegatedPermissionGrant.ReadWrite.All as an application permission.
  • 4If the SP does not have this permission → 403 Insufficient privileges and apply fails.
  • 5The permission cannot be self-granted — a Global Admin or Privileged Role Admin must assign it once.
Why it re-triggers after successful runs
If the azuread_application resource is modified (e.g. redirect URI change), Terraform may destroy and recreate the delegated permission grant. If the SP's Graph permission was revoked in the meantime, the apply will fail again.
The Fix (one-time, run as Global Admin)
# Step 1 — find the CICD service principal SP_OID=$(az ad sp show \ --display-name "azure-penny-github-actions" \ --query id -o tsv) # Step 2 — find Microsoft Graph SP in this tenant GRAPH_OID=$(az ad sp show \ --id 00000003-0000-0000-c000-000000000000 \ --query id -o tsv) # Step 3 — verify the correct app role ID az ad sp show \ --id 00000003-0000-0000-c000-000000000000 \ --query "appRoles[?value==\ 'DelegatedPermissionGrant.ReadWrite.All'].id" -o tsv # → 8e8e4742-1d95-4f68-9d56-6ee75648c72a # Step 4 — assign the app role az rest --method POST \ --uri "https://graph.microsoft.com/v1.0/\ servicePrincipals/$SP_OID/appRoleAssignments" \ --headers "Content-Type=application/json" \ --body "{ \"principalId\":\"$SP_OID\", \"resourceId\":\"$GRAPH_OID\", \"appRoleId\":\"8e8e4742-1d95-4f68-9d56-6ee75648c72a\" }"
Note: az ad app permission admin-consent does NOT work for application permissions (Role type) — it returns Consent validation failed. Always use the az rest approach above.
Bootstrap Checklist — New Environment
Everything needed before terraform apply will succeed end-to-end
Manual one-time steps (require Global Admin or Owner)
  • 1 Create the CICD app registration and service principal in Entra ID.
  • 2 Add a federated credential for repo:mysak7/azure-penny:ref:refs/heads/main (and optionally for pull_request).
  • 3 Assign Azure RBAC roles to the CICD SP: Contributor + User Access Administrator at subscription scope.
  • 4 Grant Graph permission DelegatedPermissionGrant.ReadWrite.All via az rest (see slide 7).
  • 5 Create Terraform state storage — run scripts/setup-backend.sh (creates RG, storage account, container).
GitHub repository configuration
  • 6 Set repository variables (not secrets): AZURE_CLIENT_ID, AZURE_TENANT_ID, AZURE_SUBSCRIPTION_ID, ACR_LOGIN_SERVER.
  • 7 Update cicd_principal_object_id in terraform/variables.tf to the new SP's object ID.
  • 8 Update owner_email in terraform/variables.tf to the user who should have dashboard access.
Then just push
git push origin main # Terraform workflow runs automatically: # → terraform plan (shows all resources to create) # → terraform apply (on main push) # All Entra ID resources created automatically

Summary
The complete Entra ID picture for azure-penny
🐙
CICD Service Principal
OIDC federated · no secrets.

Needs Contributor + User Access Administrator on the subscription, plus DelegatedPermissionGrant.ReadWrite.All on Graph.

Graph permission = one-time manual bootstrap.
🔑
Managed Identity
Created by Terraform. Attached to the Container App.

Holds three RBAC roles: Storage Blob Data Reader, AcrPull, and Contributor.

DefaultAzureCredential uses it automatically — no code differences between local and production.
🛡️
Easy Auth App Registration
Created by Terraform. Gatekeeps the dashboard UI.

app_role_assignment_required = true — whitelist model.

Admin consent pre-granted via azuread_service_principal_delegated_permission_grant — requires CICD SP to hold the Graph permission.
No stored passwords anywhere Fully Terraform-managed after bootstrap Bootstrap ≈ 5 az CLI commands See docs/entra-id-setup.md for full reference

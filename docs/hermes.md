# Project: Penny (azure-penny / aws-penny)
- **Business Domain:** FinOps / Cloud Cost Management Dashboard
- **Core Architecture:** FastAPI application that reads Azure Cost Management daily CSV/Parquet exports from Azure Blob Storage into a pandas DataFrame held in-memory, then serves aggregated cost data through a REST JSON API and Jinja2-rendered HTML dashboards. Data flows from Azure Cost Management → Blob Storage (`cost-exports` container) → `_discover_latest_blobs()` → pandas in-memory cache (TTL 1 h) → FastAPI routes; a second real-time path queries Azure ARM, Compute, and Retail Prices APIs to surface live resource inventory with per-resource monthly cost estimates (TTL 15 min). A third path is an **AI chat agent** (`ai_chat.py: _run_ai_chat`): an agentic tool-use loop (max 5 iterations) that calls internal cost-query tools, collects results, and returns the final answer as a string; the web SSE endpoint (`routers/ai.py`) streams it word-by-word to the browser while the Telegram handler (`telegram_bot.py`) sends it as an HTML message. A fourth path is **Shield** (`shield.py`): an asyncio background loop that runs every 15 minutes, sums projected monthly cost from live ARM resources, and fires a Telegram breach alert when the total exceeds a configurable threshold (2-hour cooldown between repeated alerts).
- **Current Version:** 1.5.0
- **Primary Tech Stack:** Python 3.11, FastAPI 0.111.0, uvicorn 0.29.0, pandas 2.2.2, pyarrow 16.0.0, azure-storage-blob 12.19.1, azure-identity 1.16.0, azure-mgmt-resource 23.1.1, azure-mgmt-compute 31.0.0, azure-mgmt-storage 21.1.0, azure-monitor-query ≥ 1.4.0, Jinja2 3.1.4, python-dotenv 1.0.1, httpx (async HTTP client for LLM proxy calls and Telegram Bot API). Parallel AWS version uses boto3 ≥ 1.34, pandas 3.0.3, pyarrow 24.0.0 (requirements differ between repos). **LLM:** Gemini 3.5 Flash via shared LiteLLM proxy at `vertex.mysak.fun` (Cloudflare Tunnel + CF Access); neither app calls Vertex AI directly.

## 1. Infrastructure & Cloud Resources (PaaS/IaaS)
- **Cloud Provider:** Azure (primary, `azure-penny`) / AWS (parallel version, `aws-penny`)
- **IaC Tool:** Terraform `hashicorp/azurerm` 4.67.0 + `hashicorp/azuread` ~3.0 + `azure/azapi` ~2.0 + `hashicorp/random` 3.8.1. Remote state stored in `azurerm` backend at storage account `azurepennytff04cd1`, container `tfstate`, keys `azure-penny.tfstate` (prd) / `azure-penny-dev.tfstate` (dev). Module structure is flat — 9 `.tf` files in `terraform/`, no submodules. AWS parallel uses `hashicorp/aws ~5.0` with an S3 backend.
- **Compute & Runtime:** Azure Container Apps — `azurerm_container_app` named `ca-{env}-{location_short}-penny`; container spec: 0.5 vCPU / 1 GiB RAM; scale rule: `min_replicas = var.always_on ? 1 : 0`, `max_replicas = 1`; `revision_mode = "Single"`. ACA Environment backed by Log Analytics workspace (SKU `PerGB2018`, 31-day retention, 1 GB/day quota cap). AWS parallel: ECS Fargate task (512 CPU units / 1024 MiB) behind an ALB with TLS termination (ELBSecurityPolicy-TLS13-1-2-2021-06).
- **Networking & Security:** ACA ingress is external on port 8000 over HTTPS; no VNet injection (standard ACA consumption plan). Easy Auth (ACA `authConfigs` resource via `azapi_resource`) enforces Entra ID authentication for all routes **except** `/telegram/webhook` (excluded via `excludedPaths` so Telegram can reach the endpoint without an Entra token); unauthenticated requests redirect to login. Role-based access within the app (`penny-admin` Entra app role) is checked by parsing the `X-MS-CLIENT-PRINCIPAL` Base64 header in `auth.py`. Admin consent pre-granted via `azuread_service_principal_delegated_permission_grant`. No private endpoints or VNet peering configured. AWS version: ALB SG (HTTP 80 → HTTPS 301, HTTPS 443), ECS SG (8000 from ALB only), default VPC.
- **Storage & Databases:** `azurerm_storage_account` (account kind `BlobStorage`, tier `Standard`, replication `LRS`, access tier `Hot`, TLS 1.2 minimum, public blob access disabled, blob versioning enabled, soft delete 7 days for blobs and containers); container `cost-exports` with `private` access. No database. Terraform state backend is a separate storage account (`azurepennytff04cd1`). AWS parallel stores CUR Parquet exports in an S3 bucket (bucket name injected via `CUR_BUCKET` env var).

## 3. Monitoring, Reporting & Day-2 Operations
- **Observability Stack:** Log Analytics workspace (`log-{env}-{location_short}-penny`, PerGB2018) receives ACA container and platform logs automatically via the ACA Environment binding (`log_analytics_workspace_id`). Application-level logging via Python `logging.basicConfig` to stdout (captured by the ACA log sink). Azure Monitor Metrics polling available per-resource via `azure-monitor-query` `MetricsQueryClient` (7 resource type metric profiles defined in `_RESOURCE_METRICS`: VMs, VMSS, Container Apps, AKS, App Service, Storage, Redis).
- **Log Forwarding:** Container stdout/stderr → ACA Environment → Log Analytics workspace. No external SIEM or event hub configured in Terraform.
- **Alerting Criteria:** No alerts configured in Terraform. The **Shield** module (`shield.py`) provides in-application alerting: a background asyncio task runs every `SHIELD_CHECK_INTERVAL` seconds (default 900 s / 15 min), sums the projected monthly cost of all live ARM resources, and sends a Telegram HTML alert when the total exceeds the configured `monthly_threshold`. A `SHIELD_ALERT_COOLDOWN` (default 7200 s / 2 h) prevents alert spam while a breach persists. The threshold and chat ID are persisted to `shield_config.json` (updated via `POST /api/shield/config` or the `/shield` dashboard page); current status is exposed at `GET /api/shield/status`. The application also surfaces anomaly data via `GET /api/anomalies`: week-over-week cost spikes > 50%, new services appearing with cost > $1, vanished services with prior-week cost > $1, and untagged resource cost totals — dashboard-only, not automated alerts.

## 4. Technical Challenges & Complex Workflows Resolved

**1. Cumulative export deduplication (`storage.py: _discover_latest_blobs`)** — Azure Cost Management daily exports are cumulative within a billing period: each run writes a new file into the same `YYYYMMDD-YYYYMMDD` folder, so naively loading all blobs would multiply every cost row by the number of runs that day. The solution lexicographically sorts date-range folder segments, walks them newest-first, and within the chosen folder selects only the single blob with the latest `last_modified` timestamp. Falls back to newest `.parquet`/`.csv` file if no date-range folder structure is detected (handles non-standard export paths). This prevents double-counting without requiring any external coordination.

**2. Multi-source cost enrichment with projection logic (`live_resources.py: _get_live_data`)** — Each live ARM resource gets its `monthly_cost` from one of six sources with explicit priority: (a) `spot_rate` — live Azure Retail Prices API call filtered for `skuName contains "Spot"` with table fallback; (b) `price_table` — on-demand retail price × 24 × 30 × instance_count; (c) `export_hours` — for VMs with < 28 days in export, derives hourly rate from `C_QUANTITY` (billing hours) then extrapolates; (d) `export_projected` — linear extrapolation from partial-month export (cost / actual_days × 30); (e) `export_last_day` — for storage accounts with Premium File Shares, picks the higher of window-projected vs. last-day-projected; (f) `export` — full 30-day window used as-is. Storage accounts whose cost is already broken out via file share inventory are zeroed out (marked `in_shares`) to avoid double-counting at the account level. Resource ID matching normalises sub-resource paths via regex (`_normalize_rid`) before joining against the billing DataFrame.

**3. Role-based access via Easy Auth header (`auth.py: _get_user_roles`, `_require_admin`)** — ACA Easy Auth injects the authenticated user's JWT claims as a Base64-encoded JSON blob in `X-MS-CLIENT-PRINCIPAL`. The `_get_user_roles` function decodes and parses this header to extract `claims[].val` where `typ == "roles"`, yielding the user's Entra app role values (`penny-admin`). The `_require_admin` FastAPI dependency gates all resource mutation endpoints (`DELETE /api/resource`, `DELETE /api/resource-group`, `DELETE /api/resource-groups/all`). When the header is absent (local dev without Easy Auth), all access is allowed through — a deliberate dev convenience that creates a potential bypass if the ACA auth middleware is ever misconfigured or bypassed.

**4. Shared AI agentic loop (`ai_chat.py: _run_ai_chat`, `_execute_chat_tool`)** — `_run_ai_chat(question, history) → str` is the single async coroutine that runs the full agentic loop and returns the final text answer. It is called by two callers: `routers/ai.py` (web SSE endpoint — streams the result word-by-word with 22 ms inter-word delay for a typewriter effect) and `telegram_bot.handle_update` (Telegram — receives full string, converts markdown to Telegram HTML via `_md_to_html`, sends via `sendMessage`). The loop: (1) appends user message to conversation, (2) POSTs to the LiteLLM proxy (`vertex.mysak.fun/v1/chat/completions`) with `tools` and `tool_choice: "auto"`, (3) if the model returns `tool_calls`, executes each locally via `_execute_chat_tool` (reads from the in-memory pandas DataFrame — no extra I/O), appends the tool result as `role: tool`, repeats up to 5 times; (4) once the model returns plain text, returns it. If 5 iterations exhaust without a final text answer, a sixth no-tools call forces one. Seven read-only tools are defined: `get_cost_summary`, `get_anomalies`, `get_resource_groups`, `get_breakdown`, `get_live_resources`, `get_daily_costs`, `get_page_snapshot` (all-in-one snapshot, preferred for broad overview questions). Parallel aws-penny uses an identical loop with `get_accounts` instead of `get_resource_groups` and no `get_page_snapshot`. See `docs/agent-architecture.md` for a full side-by-side comparison.

**5. Shield alert loop (`shield.py: shield_check_loop`, `run_shield_check`)** — `shield_check_loop(get_live_data_fn)` is an infinite asyncio task started in `lifespan`. It sleeps for `SHIELD_CHECK_INTERVAL` seconds, then calls `run_shield_check` which: loads config from `shield_config.json`, calls `get_live_data_fn()` to get current live resources, sums `monthly_cost`, compares to threshold. On breach: checks cooldown (skips if within `SHIELD_ALERT_COOLDOWN`), builds an HTML Telegram message via `_build_alert_text` (header, projected total, threshold, % over, top-10 resources), POSTs to `https://api.telegram.org/bot{TOKEN}/sendMessage`, then persists the updated `last_alerted_at` timestamp to `shield_config.json`.

**6. Telegram webhook security and routing (`telegram_bot.py`)** — Telegram webhook calls arrive at `POST /telegram/webhook`. The route is excluded from ACA Easy Auth (via `excludedPaths` in `auth.tf`) so Telegram's servers can reach it without an Entra ID token. Every call is validated against the `X-Telegram-Bot-Api-Secret-Token` header (set at `setWebhook` time, value from `TELEGRAM_WEBHOOK_SECRET`). Messages from any chat other than the whitelisted `TELEGRAM_CHAT_ID` are silently ignored. Per-chat conversation history is held in `_tg_history: dict[int, list[dict]]` (in-process, capped at 20 messages, cleared on restart).

## 5. Potential Bottlenecks & Technical Debt

- **Single-replica, no HA** — `max_replicas = 1` is hardcoded; there is no horizontal scaling or blue/green strategy. Any ACA revision deployment causes a brief cold-start gap. The `min_replicas = 0` (scale-to-zero) option adds up to ~30 s cold-start latency on first request after idle, which includes downloading the ~500 MB Python image layer.
- **Synchronous ARM SDK calls blocking the thread pool** — All Azure SDK calls (ARM inventory, Compute API, Storage Management, Monitor query, Retail Prices API) are wrapped in `run_in_executor(None, ...)` using the default ThreadPoolExecutor. The ARM inventory fetch (`_fetch_resource_inventory`) makes one `resources.list()` call plus N individual `virtual_machines.get(..., expand="instanceView")` calls (one per VM) plus NIC lookups — all sequential within a single thread pool task. For subscriptions with many VMs this can take tens of seconds, blocking that executor slot.
- **Unbounded pandas in-memory footprint** — The full cost export DataFrame is held in a single process-level dict without size limits. Azure Cost Management CSV exports for large subscriptions can reach hundreds of MB; at 1 Gi container memory with pandas overhead (typically 3-5× raw file size) this can cause OOM kills. No streaming or chunked read is implemented.
- **No test suite** — CI runs only ruff linting and `terraform validate`. There are no unit tests, no integration tests, and no mocking of Azure SDK calls, making refactoring risky.
- **Hardcoded API version fallback table (`routers/infra.py: _get_api_version`)** — The resource deletion endpoint maintains a static dict of ARM API versions per resource namespace (e.g., `"microsoft.compute": "2024-03-01"`). These will drift out of date as Azure deprecates API versions, causing silent fallback to potentially unsupported versions.
- **`always_on = true` default** — The variable default keeps one replica running at all times (saves cold start, costs ~$30-35/month on ACA consumption). The cost dashboard ironically incurs unnecessary cost if usage is low.
- **No Cost Management export automation** — `cost_export.tf` was removed from Terraform state (`removed` block with `destroy = false`) because MCA billing-scope RBAC cannot be managed via the REST API for MSA-owned accounts. The export must be created manually in the Azure Portal; there is no automation or documentation check to verify it is configured.
- **`shared_access_key_enabled = true` on storage account** — Comment notes this is a "break-glass" setting, but it means storage account keys exist and could be used to bypass the managed identity auth model if leaked.
- **Shield config persisted to local file** — `shield_config.json` is written to the container filesystem next to the module. In ACA (no persistent volume) this is lost on every revision restart; the threshold must be re-set after each deployment. A better approach would be Azure App Configuration or a Key Vault secret.
- **Simulated token streaming** — The typewriter effect in `routers/ai.py` is achieved by splitting the completed LLM response on spaces and yielding each word with `asyncio.sleep(0.022)`. This is not real token streaming — the full response is buffered before the first word is sent. If the LLM call takes 10 s, the user sees a 10 s blank followed by a fast typewriter. True streaming would require `stream: true` on the proxy call and SSE forwarding of partial chunks.
- **Single shared LLM proxy is a SPOF** — Both azure-penny and aws-penny route all LLM traffic through `vertex.mysak.fun`. A proxy outage, Cloudflare Tunnel disruption, or LiteLLM restart disables the chat widget and Telegram bot in both apps simultaneously. There is no fallback model or retry with a different endpoint configured.
- **Chat history unbounded in client** — The `history` field is accepted from the client and trimmed to the last 20 messages (`[-20:]`) server-side. A client could send a crafted large history to inflate token usage; there is no server-side token counting or hard message-size limit.
- **Telegram history lost on restart** — `_tg_history` is an in-process dict; all per-chat conversation context is lost whenever the container restarts or a new ACA revision is deployed.

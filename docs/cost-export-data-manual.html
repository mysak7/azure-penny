Cost Export Data Manual

How azure-penny loads, processes, and serves historical cost data from Azure Cost Management CSV exports

Introduction

The Manager and Technician pages display historical billing data from Azure Cost Management exports. This is entirely separate from the live ARM inventory — no Azure API calls are made during a cost query; all aggregation runs in-process against a pandas DataFrame that was loaded once from Blob Storage and cached in memory.

The result is a fast, read-only cost dashboard that answers questions like: how much did we spend on Compute this week? Which resource group is the biggest cost driver? What does our daily spend trend look like?

Not covered here: Live resource inventory and cost projections (ARM API, spot prices, file share estimates) are documented in the Live Cost Data Manual.

End-to-End Pipeline

Every dashboard API request follows the same path:

Azure Cost Management
Daily CSV export → Blob Storage → _discover_latest_blobs()
find newest date-folder → _read_blob_to_bytes()
stream download → _blob_to_dataframe()
parse CSV / Parquet → _apply_column_map()
normalize to C_* → 1h in-memory cache → filter: period + category + RG → groupby + sum → JSON

Steps 1–5 run only on a cache miss (first request, or after TTL expiry / manual reload). Steps 6–8 run on every API call, in memory, in microseconds.

Auth & Blob Client

storage.py:get_blob_service_client() returns a lazily-initialised singleton BlobServiceClient. It uses DefaultAzureCredential from the azure-identity SDK. 

credential = DefaultAzureCredential(managed_identity_client_id=AZURE_CLIENT_ID or None)

The client is a module-level singleton — one instance per process lifetime. Because Container Apps run a single replica, this means one client for the entire application. SDK connection pooling is handled internally.

Export Timing & Timezone

Timezone: always UTC

Azure Cost Management operates entirely in UTC. The Terraform schedule is anchored to 2026-05-04T00:00:00Z and runs daily with no offset option — Azure does not allow you to pick a specific time of day for daily exports. All dates in the exported CSV (Date column) are also in UTC.

When does a new export file appear in Blob Storage?

Azure runs daily exports at a time it chooses, typically somewhere between midnight and 04:00 UTC. The exact minute varies day to day and cannot be controlled. After the export process completes, the file is written to the container and becomes visible to the app on the next cache miss.

UTC timeWhat happens
00:00Azure closes billing for the previous calendar day (Day N−1)
00:00 – 04:00Azure Cost Management processes charges and generates the MonthToDate CSV
~01:00 – 04:00New CSV file lands in cost-exports/penny/YYYYMMDD-YYYYMMDD/{guid}/
after file landsNext app cache miss picks up the new file automatically
Practical rule of thumb: If you open the dashboard after 04:00 UTC, yesterday's charges should be present. Before 04:00 UTC, the dashboard may still show data from the day before yesterday. The data_as_of field in every API response tells you exactly which day is the most recent in the loaded file — use GET /api/status or GET /api/debug to check it.

Data lag: how old is the data?

Azure Cost Management charges have an inherent lag — they are not real-time. Charges for a given day are typically finalised and included in the export 24–48 hours later. In some cases (marketplace charges, reservation adjustments) the lag can reach 72 hours.

Today (UTC)Most recent data you can expectWhat data_as_of shows
Monday 06:00Saturday's charges (best case Sunday)2026-05-03 or 2026-05-02
Monday 02:00Friday's or Saturday's charges (export may not have run yet)2026-05-02 or 2026-05-01
1st of month 06:00Last 2 days of previous month (new month export just started)2026-04-29 typical

What the app does when the export is missing or stale

  • If today's folder has no file yet, _discover_latest_blobs() moves to the previous date folder and uses that file instead — silently.
  • The data_as_of field will be 1–2 days behind today, which is expected and normal.
  • If no export file exists at all (e.g. first deployment before the first scheduled run), the app returns an empty DataFrame and all API endpoints return zero/empty results with no error.
  • You can trigger a one-time manual export from the Azure portal (Cost Management → Exports → Run now) to seed the container immediately without waiting for the overnight job.

Your local time vs. the dashboard

All dates in the dashboard are the raw YYYY-MM-DD values from the export, which are UTC calendar dates. If you are in a UTC+2 timezone and it is 01:00 local time on Tuesday, it is still Monday 23:00 UTC — the dashboard will show Monday as the most recent day, not Tuesday.

No timezone conversion is performed. The app passes the C_DATE strings straight through from the CSV to the API response. What you see is always UTC date.

Blob Discovery

_discover_latest_blobs(container_name) lists all blobs in the container and selects exactly one file to load. Azure Cost Management exports are cumulative — each daily file contains all charges from the start of the billing month, so loading only the newest file gives the complete dataset.

Primary algorithm (date-folder pattern)

Azure exports are stored in penny/{YYYYMMDD-YYYYMMDD}/{guid}/filename.csv paths. The function extracts the YYYYMMDD-YYYYMMDD segments and sorts them descending.

list_blobs() → extract all YYYYMMDD-YYYYMMDD
path segments → sort descending
(newest first) → for each folder: find
.parquet/.csv/.csv.gz → if multiple files:
pick newest by last_modified → return [blob_name]

Fallback algorithm (no date folders)

If no YYYYMMDD-YYYYMMDD segments are found (e.g. a manually uploaded file or a non-standard path), the function falls back to selecting the single newest .parquet, .csv, or .csv.gz file in the entire container, by last_modified timestamp.

Why only one file?

If two files from different date folders were concatenated, costs would be double-counted. A June 1–5 file and a June 1–10 file both contain June 1–5 charges — stacking them inflates every figure by 2×. The latest folder always has the most complete data for the billing period.

What if today's export hasn't run yet? The app will silently fall back to the previous day's export. The data_as_of field in every API response tells you the last date present in the loaded file. If it's two days old, the morning export hasn't finished yet.

File Reading

After discovery, _read_blob_to_bytes() streams the blob into a Python bytes object entirely in memory — no temporary disk file is written. The raw bytes are wrapped in io.BytesIO and handed to pandas.

File extensionpandas callNotes
.parquet pd.read_parquet(buf, engine="pyarrow") Fastest; preserves dtypes. Azure Cost Management doesn't currently export Parquet at billing-profile scope, but the app supports it for future compatibility.
.csv.gz pd.read_csv(buf, compression="gzip") Gzip-compressed CSV. Smaller download but slower to parse.
.csv pd.read_csv(buf) Standard CSV. Azure currently exports in this format at billing-profile scope.

Multiple files from the same folder are supported (returns a list), concatenated via pd.concat(frames, ignore_index=True). In practice, Azure always writes exactly one file per export run.

Mapping algorithm

The mapping is case-insensitive. For each raw column:

  1. Lowercase the raw column name.
  2. Look it up in COLUMN_MAP (keys also lowercased).
  3. If found and the internal name is not already taken, add to the rename dict.
  4. If the column starts with tag_ and C_TAGS isn't taken, map it to C_TAGS.

After renaming:

  • C_COST → pd.to_numeric(..., errors="coerce").fillna(0.0) (strings / blanks become 0)
  • C_DATE → parsed as datetime, then formatted to YYYY-MM-DD string
  • C_NAME → lowercased (resource group names are compared case-insensitively throughout)

Missing columns

If a required internal column (C_COST, C_SERVICE, C_NAME, C_ACCOUNT, C_DATE) is not found after mapping, a warning is logged but the app continues with whatever columns are available. API endpoints that depend on a missing column will return empty results rather than an error.

In-Memory Cache

After parsing, the DataFrame is stored in a module-level dict (_cache in storage.py). The async lock (_lock) prevents concurrent reloads from racing. 

# storage.py
_lock:  asyncio.Lock = asyncio.Lock()
_cache: dict[str, Any] = {}   # keys: "df", "loaded_at"

async def get_cached_dataframe() -> pd.DataFrame:
    async with _lock:
        now = time.monotonic()
        if "df" not in _cache or (now - _cache.get("loaded_at", 0.0)) > TTL_SECONDS:
            df = await asyncio.get_event_loop().run_in_executor(None, _load_dataframe)
            _cache["df"]        = df
            _cache["loaded_at"] = now
        return _cache["df"]
PropertyValue
TTLCACHE_TTL_SECONDS env var, default 3600 s (1 hour)
Concurrencyasyncio.Lock — only one thread loads at a time; others wait at the lock
ExecutorThe blocking SDK download runs in run_in_executor(None, ...) (default thread pool) to avoid stalling the event loop
Cold-start behaviourFirst request triggers a full download + parse. Typical duration: 2–10 s depending on file size. All other requests during that window queue at the lock and receive the same result.
Error behaviourIf no export files are found, an empty DataFrame with the required column schema is cached. APIs return empty/zero results instead of 500 errors.
Manual invalidationPOST /api/reload or GET /costs/refresh — clears _cache, then immediately re-loads
Scale-to-zero note: When the Container App is idle and scales down to zero replicas, the process terminates and the cache is lost. The next request after a cold start always re-downloads from Blob Storage, regardless of when it was last loaded.

Period System

Every category API accepts a period query parameter. The parameter maps to a number of trailing calendar days, measured backward from today.

period=day
1 day
Yesterday's charges (or latest available day — see Day Fallback)
period=week
7 days
Default. Last 7 calendar days.
period=month
30 days
Last 30 calendar days.

The cutoff date is computed at request time:

cutoff = (date.today() - timedelta(days=days)).strftime("%Y-%m-%d")
filtered = df[df["C_DATE"] >= cutoff]

Because the export always contains month-to-date data, requesting period=month when only 5 days into the month will return only 5 days of charges — not a projected 30. The data_as_of field in the response tells you the most recent date in the filtered result.

Service Categories

Category endpoints filter the DataFrame to a named set of C_SERVICE values (i.e. MeterCategory from the export). The sets are defined as Python frozen sets in main.py.

Matching is case-insensitive. Both the export's C_SERVICE values and the set members are lowercased before comparison, so Virtual Machines matches virtual machines in the export. Services not in any category set appear only in /api/services (the "All Services" view) — they are not lost.

Filter Helpers

Four helper functions in main.py compose to build every API response:

FunctionSignatureWhat it does
_filter_period(df, days) (DataFrame, int) → DataFrame Keeps only rows where C_DATE ≥ today − days. No-op if C_DATE column is absent.
_filter_services(df, services) (DataFrame, set[str]) → DataFrame Keeps rows where C_SERVICE.lower() is in the lowercased services set. No-op if C_SERVICE is absent.
_filter_rg(df, rg) (DataFrame, str) → DataFrame Keeps rows where C_NAME == rg.lower(). No-op if rg is empty or C_NAME is absent.
_cost_by(df, col) (DataFrame, str) → dict[str, float] Groups by col, sums C_COST, filters out zero-cost groups, sorts descending. Returns {name: cost}.

Typical composition (category API)

full_df   = await get_cached_dataframe()
df        = _filter_rg(full_df, rg)          # optional RG scope
filtered  = _filter_services(
              _filter_period(df, days),      # time window
              CAT_COMPUTE                    # service set
            )
by_svc    = _cost_by(filtered, "C_SERVICE")  # aggregate

Day Fallback

When period=day is requested and the filtered DataFrame is empty (today's export hasn't run yet, so there's nothing newer than yesterday), the app automatically falls back to the most recent date available in the full dataset. 

if period == "day" and filtered.empty and "C_DATE" in full_df.columns:
    last_date = full_df["C_DATE"].dropna().max()
    filtered  = df_svc[df_svc["C_DATE"] == last_date]
    fallback  = True

When fallback is active, the data_as_of field in the response contains the actual date used, and the UI badge changes from e.g. "24 hours" to "Latest: 2026-05-03" so the user can see they're looking at older data.

Storage Breakdown: Space vs. Transfer

/api/storage/breakdown splits storage cost into two buckets: Space/Capacity (data stored, disks, backups) and Transfer/Bandwidth (egress, replication, retrieval).

The classification uses keyword matching on the combined C_SERVICE + C_SUBCATEGORY string: 

_TRANSFER_KEYWORDS = frozenset([
    "bandwidth", "transfer", "egress",
    "geo-redundant replication", "data retrieval", "replication",
])

def _row_type(svc: str, sub: str) -> str:
    combined = (svc + " " + sub).lower()
    return "transfer" if any(k in combined for k in _TRANSFER_KEYWORDS) else "space"

Bandwidth is included in the storage query for this endpoint (even though it's also part of CAT_NETWORK) because bandwidth charges in Azure are tightly associated with storage egress and make the breakdown useful.

Services & Daily Endpoints

GET /api/services All services — no category filter, full MeterCategory breakdown

Parameters: period=week rg=

Returns: services sorted by cost, total_usd, data_as_of

Unlike the category endpoints, this groups the entire filtered DataFrame by C_SERVICE with no service-set filter. It shows every MeterCategory value that appears in the export — including services that don't fall into Compute, Storage, Network, or Database.

This is the source for:

  • The Technician "All Services" tab
  • The status-bar grand total on both Manager and Technician pages
  • The Insights tab (top spender, share analysis)
  • The Insights "Category Breakdown" sub-card (calls all four category endpoints)
GET /api/daily Daily spend timeseries for the last N days

Parameters: days=30 (default 30) rg=

Returns: points array of {date, cost_usd} sorted by date ascending, total_usd

Groups the full DataFrame (no service filter) by C_DATE and sums C_COST. Used to draw the bar chart and last-10-days table on the Technician "All Services" tab.

The canvas chart colours weekend bars (getUTCDay() === 0 or 6) in a slightly darker blue than weekdays.

Resource Group Endpoints

GET /api/resource-groups Total cost aggregated by resource group for a given period

Parameters: period=week rg=

Returns: resource_groups array of {name, cost_usd} sorted by cost, total_usd

Groups the full period-filtered DataFrame by C_NAME (resource group). No service filter — all spend regardless of category.

Used by Technician → Resource Groups tab. Each row has a delete button that triggers the RG deletion modal.

GET /api/resource-groups-list Resource group names with their all-time total cost; used to populate the RG dropdown

No parameters.

Returns: resource_groups array of {name, total_cost}

Unlike /api/resource-groups, this uses the entire DataFrame (no period filter) to build the dropdown list, so every resource group that ever appeared in exports is shown, even if it had no spend in the current period.

ARM fallback: If no cost export data is loaded yet (empty DataFrame), it falls back to calling list_resource_groups() via the ARM API and returns {name, total_cost: null} for each. In this state the dropdown works but costs are not shown.

Raw & Utility Endpoints

GET /costs Full aggregated DataFrame as JSON — no period filter

Groups by [C_SERVICE, C_NAME, C_DATE] and sums C_COST. Returns every row in the loaded export. The response can be large for a full month's export. Intended for debugging and custom analysis, not UI display.

GET /costs/summary Top 5 services and resource groups across all loaded data

Returns: total_cost, top_services (top 5), top_resource_groups (top 5)

No period or RG filter. Useful as a quick overview of the entire loaded export.

GET /api/status Cache metadata — shown in the status bar on every page

Returns:

  • storage_account — configured storage account name
  • container — blob container name
  • row_count — total rows in the cached DataFrame
  • date_range — [min_date, max_date] of C_DATE column
  • cache_age_s — seconds since the DataFrame was last loaded
  • no_data — true if the DataFrame is loaded but empty

The UI polls this on page load to populate the status bar (Date range, Rows, Cache age).

POST /api/reload Invalidate the cost export cache and re-download from Blob Storage

Clears _cache dict, then immediately calls get_cached_dataframe() to trigger a fresh download. Returns {"status": "reloaded", "rows_loaded": N}.

Also available as GET /costs/refresh which does the same thing (kept for convenience).

Triggered by the ↻ Reload data button in the status bar on all pages.

GET /api/debug Full diagnostic snapshot — blob listing, column names, date range, top services

Returns:

  • blob_count — total blobs in the container
  • blob_paths_sample — first 20 blob paths (for verifying folder structure)
  • row_count — DataFrame rows
  • columns — all column names after mapping
  • date_range — {min, max} of C_DATE
  • total_cost — sum of all C_COST
  • top_services — top 20 distinct C_SERVICE values by frequency
  • top_resource_groups — top 10 distinct C_NAME values by frequency

The most useful endpoint when the dashboard shows no data — check columns to verify column mapping worked, and blob_paths_sample to verify the export landed in the expected path.

Shared Controls Across Pages

Period pills (Day / Week / Month)

Present on both Manager and Technician. Default is Week. Changing the period sets loaded = {} and re-fires all API calls. The active period is not persisted between page navigations — it resets to Week on each page load.

Resource Group dropdown

A dropdown in the page header (centred, absolutely positioned). Populated from /api/resource-groups-list. Selection is stored in localStorage under the key az_penny_rg and restored on every page load, including the Live page — so the RG filter is shared across all three pages.

  • When a RG is selected, all API calls append &rg={name}.
  • The dropdown border changes to blue when a filter is active (class="rg-select filtered").
  • Select All to clear the filter and remove from localStorage.

Status bar

Loaded via GET /api/status on page load. Shows:

  • Source: always "Cost Management / Blob Storage" (hardcoded label)
  • Total: grand total from /api/services for the active period
  • Date range: min and max C_DATE in the cached DataFrame
  • Rows: total row count, formatted with K/M/B suffix
  • Cache age: seconds since last load, shown as minutes

Reload button

The ↻ Reload data button in the status bar calls POST /api/reload. On success, it clears the client-side loaded map and re-fires all visible API calls. Use this after a new cost export has landed in Blob Storage to pick up today's data without waiting for the TTL to expire.

Data-as-of badge

Each category card has a badge that updates after the API responds. Normally it shows the period length (e.g. "7 days · as of 2026-05-03"). When day-fallback activates it shows "Latest: 2026-05-02" to signal that today's data isn't available yet.

Reloading & TTL

When does a reload happen?

TriggerMechanism
Cache miss on startup or after scale-from-zeroAutomatic — first API call loads the DataFrame
TTL expiryAutomatic — next API call after CACHE_TTL_SECONDS triggers a reload
User clicks ↻ Reload dataManual — POST /api/reload clears _cache and immediately re-downloads
GET /costs/refreshSame as above, GET variant

When should you manually reload?

  • You just uploaded or modified a cost export file in Blob Storage.
  • Today's Azure Cost Management export has finished running (usually by 02:00 UTC) and you want to see it without waiting up to 1 hour.
  • The status bar shows a date range that doesn't include recent data.

What a reload does NOT do

/api/reload only flushes the export DataFrame cache (_cache in storage.py). It does not flush the ARM inventory cache (_live_cache in live_resources.py). To refresh live resource data, use POST /api/live-reload or the Refresh inventory button on the Live page.

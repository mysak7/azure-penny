System at a Glance
What azure-penny is and what it does

Cost Dashboard
FastAPI application serving three HTML dashboards (Dashboard, Technician, Live) that visualise Azure cost exports and live resource inventory.

Serverless Hosting
Azure Container Apps with scale-to-zero (0→1 replicas). Single revision, 0.5 vCPU / 1 GiB RAM. No baseline compute cost when idle.
🔐
Zero-Secret Auth
User-Assigned Managed Identity for all Azure SDK calls. GitHub Actions uses OIDC federation. Easy Auth (Entra ID) guards the public URL.
📁
Data Pipeline
Cost Management exports daily CSVs to Blob Storage. App discovers the latest date-range folder, reads into pandas, caches 1 hour in memory.
🔭
Live Inventory
ARM SDK enumerates all resources, fetches VM power states, correlates with cost export data, projects monthly spend per resource.
⚙️
Full IaC
100% Terraform-managed: Resource Group, Storage, ACR, ACA Environment, Budget Alert, Entra App, Easy Auth — with remote state in Blob.

Strengths — What's Done Well
Design decisions that are genuinely solid
🔑 Zero-secret authentication Best practice
Managed Identity for all Azure SDK access. GitHub OIDC federation eliminates stored secrets. No client secrets in CI/CD variables. Easy Auth guards the public endpoint without touching application code.
📦 100% Terraform-managed IaC Reproducible
Every resource — including Entra app registration, Easy Auth config (via azapi), cost export, and budget alert — is in state. Remote state in Blob Storage. Naming convention is consistent and environment-parameterised.
💤 Scale-to-zero serverless Cost-efficient
0 min replicas means zero compute cost when idle. A dashboard opened 1–2 times per day tolerates a cold start. LRS storage (no geo-redundancy) is correct since exports are rebuilt daily.
🔄 Smart blob discovery Correct
Picks only the single most-recently-modified file per date-range folder — correctly avoiding double-counting from Azure's cumulative MonthToDate export pattern. Fallback chain (parquet → csv → csv.gz) handles format changes.
⚡ Async + executor pattern Non-blocking
FastAPI routes are fully async. Blocking I/O (pandas reads, ARM SDK calls) runs in thread pool via run_in_executor(). asyncio.Lock protects the shared in-memory cache from stampede on cold start.
💰 Multi-layer cost attribution Resilient
Five fallback strategies: export match → period interpolation → projection → spot API → hardcoded table. Resources without export data still get a cost estimate. Source attribution field tells the UI which method was used.
Weaknesses — What Should Be Improved
Real risks and technical debt worth addressing
🗑️ Destructive DELETE endpoints exposed High risk
DELETE /api/resource, /api/resource-group, and /api/resource-groups/all can wipe subscription resources. They are protected only by Easy Auth (login ≠ authorisation). Any authenticated user can delete any resource group, including the one hosting the app itself.
💀 Managed Identity has Contributor on subscription Overprivileged
The identity needs Contributor at subscription scope to list and delete resources for the Live tab. This violates least-privilege. A compromised container can delete any subscription resource and is a lateral movement path.
💾 In-process in-memory cache only Fragile
Cache lives in the single container process. A container restart (common after scale-to-zero) flushes it. If max replicas ever grows to 2+, each instance builds its own cache independently — no data consistency.
🔢 Hardcoded VM pricing tables Stale risk
Spot and on-demand prices for ~5 VM SKUs are hardcoded in live_resources.py. Azure changes these without notice. Only a handful of regions and SKUs are covered — everything else falls through to an 8-second external API call or returns $0.
📋 Cost export format: CSV, not Parquet Performance
Azure Cost Management exports at subscription scope produce CSV only. Parquet is available only at billing-account scope. Monthly exports can reach 100 MB+ as CSV; pandas reads and parses the whole file on every cache miss.
🚨 No observability beyond logs Blind spots
Log Analytics workspace exists but only captures container stdout. No distributed tracing, no metrics (request latency, cache hit rate, blob read time), no alerting on application errors. Cold-start duration is invisible.
Security Analysis
Authentication model, privilege boundaries, and attack surface
What's correct
  • ✅Managed Identity — no credentials in environment or code
  • ✅Easy Auth at the ACA ingress layer — app never sees unauthenticated requests
  • ✅ACR Admin user disabled — pull only via role assignment
  • ✅GitHub OIDC federation — no AZURE_CLIENT_SECRET in repository secrets
  • ✅Storage Account: allow_nested_items_to_be_public = false
  • ✅TLS 1.2 minimum enforced on storage account
  • ✅Single-tenant Entra app — no cross-tenant login possible
Gaps & risks
  • 🔴No authorisation model — Easy Auth is authentication only; all authenticated users have equal write/delete access
  • 🔴Contributor scope too broad — identity can modify billing, networking, and security resources
  • 🟡App Password expires 2028-01-01 — no rotation mechanism; silent expiry will break Easy Auth
  • 🟡Shared access key enabled on storage — intended as break-glass but widens the blast radius if leaked
  • 🟡No IP allowlist on ACA ingress — any internet address can reach the Easy Auth login screen

Best path forward: Option B — add a single Azure Function with a Blob trigger that runs when Cost Management drops a new file. Live Resource Inventory: Alternatives
How to get real-time resource + cost data without Contributor on subscription
Current design problems
  • 🔴Contributor on subscription — required for list + delete, but massively overprivileged for read-only scenarios
  • 🔴Direct ARM calls from app — 15-min cache, but bursts of 10–20 API calls per live-reload hit ARM rate limits
  • 🟡Hardcoded VM prices — only 5 SKUs, breaks on new instance types or region launches
  • 🟡File share pricing via Storage SDK — optional dependency, fails silently
Alternative: Azure Resource Graph
Replace ResourceManagementClient.resources.list() with a single Azure Resource Graph KQL query. One API call returns all resources with type, location, RG, and tags. Requires only Reader at subscription scope. Dramatically fewer ARM API calls, no pagination issues.
Resources | where type != 'microsoft.resources/subscriptions'
| project id, name, type, resourceGroup, location

⚡ Cold start mitigation
Add a startup event (FastAPI lifespan) that pre-loads the DataFrame cache immediately on boot. Reduces first-request time from ~15 s (boot + CSV parse) to ~3 s (boot only).
🕐 Scheduled warm-up
A Logic App or GitHub Action that pings /health every 10 minutes during business hours keeps the container warm for zero extra cost. Scale stays at 0 outside hours.
📈 Scaling rule improvement
Current HTTP rule fires at 10 concurrent requests. A personal dashboard rarely reaches that. Removing the HTTP rule and relying only on KEDA HTTP scaler prevents unnecessary replica creation.
Observability Gap
What you can't see today — and what to add
Current state
  • ✅Log Analytics workspace captures container stdout/stderr
  • ✅ACA revision-level HTTP metrics (basic)
  • ✅Budget alerts (email at 20/40/60/80/100% of $100)
  • 🔴No request latency histograms
  • 🔴No cache hit/miss ratio
  • 🔴No blob read duration tracking
  • 🔴No alerting on application errors (5xx)
  • 🔴No tracing across ARM API calls
  • 🟡Cold-start duration invisible — only seen in ACA provision logs
Recommended additions
  • 1 Application Insights (free tier) — add opencensus-ext-azure or opentelemetry-sdk with Azure Monitor exporter. Gets distributed tracing, latency histograms, and dependency tracking for ~$0.
  • 2 Structured logging — switch to JSON log lines with request_id, endpoint, duration_ms, cache_hit fields. Enables KQL queries on Log Analytics without code changes in Terraform.
  • 3 /health liveness + readiness — current /health is status-only. Add liveness_probe and readiness_probe to the ACA container spec so Azure restarts unhealthy containers automatically.
  • 4 Alert rule on 5xx rate — Log Analytics alert: if ContainerAppConsoleLogs | where Log contains "500" > 3 in 5 min, fire email via existing action group.
  • 5 Cost export lag alert — alert if no new blob appears in cost-exports/penny/ within 36 hours. Catches Cost Management export silently stopping.
    • Terraform Code Quality
    IaC design: what works, what to tighten
    ✅ Consistent naming convention
    All resources follow {prefix}-{env}-{location_short}-{suffix}. Storage and ACR names correctly strip hyphens (Azure naming constraints). Random suffix on storage avoids global name collisions.
    ✅ Import block for pre-existing resources
    imports.tf brings manually-created resources under IaC control — correct approach. Avoids drift between actual Azure state and Terraform state without destructive re-creation.
    ✅ azapi for Easy Auth
    Using AzAPI provider for the Microsoft.App/containerApps/authConfigs resource is the right call — azurerm doesn't expose this resource yet. Keeps infrastructure declarative rather than using null_resource + az CLI.
    ✅ Multi-provider pattern (azurerm + azuread + azapi)
    Appropriate use of specialised providers. azuread for Entra objects, azapi for preview resources, azurerm for stable resources. Versions are pinned, preventing accidental upgrades.
    ⚠️ container_image is a required variable
    Terraform apply always requires -var container_image=.... This creates a chicken-and-egg problem: infrastructure must exist before CI/CD can push an image, but apply needs an image tag. Fix: use a placeholder default (mcr.microsoft.com/azuredocs/containerapps-helloworld:latest) and let CI/CD override.
    ⚠️ No environment separation
    All resources are prd. No dev workspace or tfvars file. Testing Terraform changes means applying directly to production. Recommend: add dev.tfvars with smaller SKUs and a separate storage container for state.
    Application Code Quality
    Python / FastAPI layer: patterns and concerns
    Strengths
    • ✅Layered modules — clean separation: main.py (routes), storage.py (blob), live_resources.py (ARM), config.py (env)
    • ✅DefaultAzureCredential — works identically in ACA (IMDS), local (CLI), CI (env vars); no conditional auth logic
    • ✅Column mapping dict — handles multiple Azure export schema variants (CostInBillingCurrency, Cost, PreTaxCost) via fallback iteration
    • ✅Fallback date period — if "day" period requested but no data, uses most recent date rather than returning empty
    • ✅Service category sets — cleanly defined as module-level constants, easy to extend
    Concerns
    • 🔴main.py is monolithic — 700+ lines: routes, filtering logic, aggregation, and ARM calls all mixed. Hard to test individual concerns.
    • 🟡No request validation — period and rg query params are used directly in filtering without sanitisation
    • 🟡No tests — no pytest, no unit coverage on filtering, aggregation, or blob discovery logic
    • 🟡Pinned to old versions — fastapi 0.111.0, uvicorn 0.29.0, pyarrow 16.0.0 (all have newer releases with bug fixes)
    Recommended refactoring
  • 1 Extract routers — split main.py into FastAPI APIRouter modules: routers/costs.py, routers/live.py, routers/admin.py. Each file is <200 lines and independently testable.
  • 2 Pydantic response models — define typed response schemas. FastAPI validates output automatically; OpenAPI docs become usable; breaking schema changes are caught at startup.
  • 3 Lifespan event for cache pre-warm — use FastAPI's lifespan context manager to start loading the DataFrame in the background immediately on container boot, before the first real request.
  • 4 Add pytest + pytest-asyncio — unit test _discover_latest_blobs() with mock blob listings, _filter_period() with synthetic DataFrames, and cost attribution logic with stub ARM responses.
  • 5 Dependency injection for storage — pass BlobServiceClient as a FastAPI dependency rather than a module-level singleton. Makes unit tests trivial to write without monkeypatching.

    Cost optimisation opportunities
    • 💡ACR Basic → ACR-less: pull images from GitHub Container Registry (ghcr.io) for free. Requires adjusting the ACA image source and OIDC authentication. Saves ~$5/mo.
    • 💡Log Analytics quota: 1 GB/day cap with 31-day retention may generate unnecessary cost. For a single-container app, 100 MB/day is more realistic. Adjust daily_quota_gb to 0.1.
    • 💡Storage tier: CSV exports are read once per hour then ignored for 23 hours. Consider moving to Cool tier after 1 day via a lifecycle policy — 50% storage cost reduction.
    • 💡ACA Environment workload profile: Dedicated profile at D4 costs ~$200/mo. Keep Consumption — correct choice for this workload.
    🏆 Bottom line on cost
    The current architecture is genuinely cost-optimised for its purpose. Scale-to-zero + LRS + Basic ACR + Consumption Log Analytics = ~$10/mo total. The only meaningful saving available is dropping ACR for GHCR ($5/mo), which introduces a trade-off in registry control. At this scale, optimise for developer experience over infrastructure cost.

    Architecture Scorecard
    Overall assessment across six dimensions
    Security & Auth 7 / 10
    Zero-secret identity ✓, but Contributor is overprivileged and no authorisation layer for destructive ops
    Infrastructure (IaC) 9 / 10
    Fully declarative, consistent naming, remote state, import blocks, multi-provider. Only missing: env separation.
    Cost Efficiency 9 / 10
    Scale-to-zero, LRS, Basic ACR, Consumption log — ~$10/mo total. Excellent for a personal cost dashboard.
    Application Code Quality 6 / 10
    Good module separation and async patterns, but monolithic main.py, no tests, hardcoded pricing, old deps.
    Observability 4 / 10
    Log Analytics exists but only captures raw stdout. No metrics, no tracing, no alerting on errors or data freshness.
    Data Pipeline 6 / 10
    Blob discovery is correct and resilient. But full CSV parsing per cold start is slow; no pre-aggregation layer.
    Overall verdict
    azure-penny is a well-designed personal cost tool that makes smart trade-offs for its scale. The IaC is production-quality. The two non-negotiable fixes are the Contributor privilege scope and the missing authorisation on DELETE endpoints — both are security issues that could cause real damage. Everything else is a quality-of-life improvement that can be done incrementally.

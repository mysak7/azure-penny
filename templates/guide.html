Data Sources
🗂 Cost Export Blobs Primary

Daily cumulative CSV/Parquet files exported by Azure Cost Management to Blob Storage. The app reads only the single newest file from the latest date-range folder to avoid double-counting.

Lag: 1–2 days (export runs 00:00–04:00 UTC)
Window: month-to-date, then filtered to last 30 days
Accuracy: Actual billed cost (EA/MCA rates applied)
☁️ Cost Management API Fallback

Used automatically when no blob exports are found in the container. Queries the /query endpoint with a 90-day window, grouped by ResourceId + MeterCategory.

Lag: Same as export (Cost Management data age)
Window: 90 days
Accuracy: Same billed cost, less granular (no tags/subscription fields)
⚡ Live ARM + Pricing API Overlay

ARM resource inventory fetched in real-time. For VMs and VMSS, the Azure Retail Prices API (or hardcoded fallback tables) provides hourly rates. Used in the Live view and to project the forecast forward.

Lag: ~0 (current to the minute)
Window: N/A — current state only
Accuracy: List price only — does not reflect RIs, AHUB, or EA discounts
Forecast & Calibration

The end-of-month forecast combines actual spend so far (from export data) with a forward projection based on live resource pricing. Because live pricing uses retail list rates, a calibration factor is applied to bring the projection in line with what the subscription actually pays.

# Step 1 — compute global calibration (subscription-level)
all_live_daily = sum(resource.monthly_cost for all resources) / 30
actual_7d_total = sum of all-RG export costs over last 7 days
raw = actual_7d_total / (all_live_daily × 7)
calibration = clamp(raw, min=0.5, max=2.0)

# Step 2 — project forward for selected RG
daily_fwd = rg_live_daily × calibration
end_of_month = spent_so_far + daily_fwd × days_remaining

Why global calibration? Hub RGs (network, shared services) have expensive ARM resources at list prices, but their actual export cost is low — billed to other RGs, or covered by reservations. A per-RG calibration collapses to the 0.5 floor for those RGs, inflating their forecast. Global calibration uses the real subscription-wide actual-vs-list ratio instead.

Clamp [0.5 – 2.0] prevents extreme swings from days with anomalous spend or very few data points. The 0.5 floor means the forecast will never project less than half of what live pricing suggests, even for heavily discounted subscriptions.

Data source is hybrid when live pricing is available, linear_fallback when the live inventory call fails (uses historical daily average instead).

Known Limitations
  • Reserved Instances & Azure Hybrid Benefit not reflected in live pricing. The Retail Prices API returns list rates. Resources covered by RIs or AHUB will show higher estimates on the Live page than what appears on the invoice.
  • Calibration floor of 0.5 can still overestimate. If the actual discount is >50% (e.g. heavy RI coverage), the calibration clamps at 0.5 and the forecast will be up to 2× the real spend.
  • Hardcoded VM price fallback covers only ~5 SKUs. If the Retail Prices API is unavailable and the VM size is not in the table, no cost is shown for that resource.
  • Export lag is 1–2 days by design. Azure Cost Management exports run between midnight and 04:00 UTC. Today's charges are never in the data.
  • Monthly cost uses a fixed 30-day multiplier (price/hr × 24 × 30), not actual days in the month. February projections will be slightly overstated.
  • Single subscription only. The app reads from one AZURE_SUBSCRIPTION_ID. Multi-subscription setups require separate deployments.
  • File share cost uses provisioned quota, not actual usage. Azure Files Premium bills for provisioned capacity, so this is correct for Premium tier but will overstate Standard tier shares.
  • In-memory cache is single-replica. If the Container App scales beyond 1 replica, each instance has its own cache and will make independent ARM/Blob calls.
